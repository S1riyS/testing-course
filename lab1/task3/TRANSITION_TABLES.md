## Задание 3 как конечные автоматы и тестирование методом таблиц переходов

Исходный текст предметной области:

> Они сидели на мостовой и смотрели с некоторым беспокойством, как огромные дети тяжело прыгают по песку, а дикие лошади с грохотом везут по небу в Неизведанные Области свежие запасы армированных изгородей.

В исходной реализации для `task3` уже есть доменная модель (`Person`, `Child`, `Horse`, `Supply` и т.д.) и модульные тесты `AppTest`. Преподаватель попросил:

- **воспринимать модель как набор конечных автоматов состояний;**
- **перепривязать/расширить тестовое покрытие, используя метод "таблицы переходов".**

Ниже показано, как это сделать именно для вашей реализации.

## 1. Конечный автомат эмоций `Person`

### 1.1. Состояния и входные воздействия

- **Состояния (Emotion):**

  - `CALM` — изначально спокоен.
  - `SOMEWHAT_WORRIED` — несколько обеспокоен.
  - `WORRIED` — явно обеспокоен.

- **События/входы:**
  - `init` — создание объекта `Person()`.
  - `watchChild(size)` — вызов `person.watch(child)` с ребёнком определённого размера.
  - `watchHorse(wild)` — вызов `person.watch(horse)` с лошадью (дикая/нет).

### 1.2. Таблица переходов состояний для `Person`

| №   | Текущее состояние | Вход / условие              | Следующее состояние | Выход / ожидаемый результат |
| --- | ----------------- | --------------------------- | ------------------- | --------------------------- |
| 1   | (нет объекта)     | `init`                      | `CALM`              | `emotion == CALM`           |
| 2   | `ANY`             | `watchChild(size = HUGE)`   | `SOMEWHAT_WORRIED`  | лёгкое беспокойство         |
| 4   | `ANY`             | `watchChild(size = NORMAL)` | `CALM`              | остаётся спокоен            |
| 5   | `ANY`             | `watchHorse(wild = true)`   | `WORRIED`           | явно обеспокоен             |
| 7   | `ANY`             | `watch(null)`               | (исключение)        | `IllegalArgumentException`  |

Фактическая логика в коде покрывает строки 1–3 и 5–7. Строки 4, 6, 8 — потенциальные расширения доменной модели и тестов.

### 1.3. Связь с существующими тестами и недостающие тесты

Связка строк таблицы с тестами `AppTest`:

- **Строка 1** — косвенно проверяется во всех тестах, где у только что созданного `Person` эмоция не меняется (можно добавить отдельный тест `personIsCalmByDefault`).
- **Строки 2–3** — покрываются тестами:
  - `watchingHugeChildCausesSomeWorry` (`theyAreOnThePavement` + `personA.watch(hugeChild)` ⇒ `SOMEWHAT_WORRIED`).
  - `worryIsSomeNotExtreme` (проверка, что не `WORRIED`).
- **Строка 7** — в текущих тестах не проверяется, стоит добавить негативный тест:
  - `watchNullThrowsException`.

**Пример новых тестов по таблице переходов (идея, не код):**

- тест на начальное состояние `CALM`,
- тест на `watchChild(size = NORMAL)` без изменения состояния,
- тест на `watchHorse(wild = true)` с переходом в `WORRIED`,
- тест на `watch(null)` с ожиданием `IllegalArgumentException`.

## 2. Конечный автомат местоположения `Person`

### 2.1. Состояния и события

- **Состояния:**

  - `NO_LOCATION` — человек создан, но ещё никуда не сел.
  - `AT_LOCATION(L)` — человек сидит на некоторой локации `L` (`Pavement`, `Sand` и т.п.).

- **События/входы:**
  - `init()` — создание объекта `Person`.
  - `sitOn(location)` — посадить человека на указанную локацию.

### 2.2. Таблица переходов для `Person` (расположение)

| №   | Текущее состояние | Вход / условие                               | Следующее состояние | Выход / ожидаемый результат                                      |
| --- | ----------------- | -------------------------------------------- | ------------------- | ---------------------------------------------------------------- |
| 1   | (нет объекта)     | `init()`                                     | `NO_LOCATION`       | `location == null`                                               |
| 2   | `NO_LOCATION`     | `sitOn(location != null)`                    | `AT_LOCATION(L)`    | `location == L`                                                  |
| 3   | `NO_LOCATION`     | `sitOn(null)`                                | (исключение)        | `IllegalArgumentException("Location cannot be null")`            |
| 4   | `AT_LOCATION(L)`  | `sitOn(location == L)`                       | (исключение)        | `IllegalArgumentException("Person is already at this location")` |
| 5   | `AT_LOCATION(L1)` | `sitOn(location = L2), L2 != null, L2 != L1` | `AT_LOCATION(L2)`   | `location == L2` (разрешённая смена локации на другую)           |

### 2.3. Связь с тестами

- Тесты `PersonLocationTest` реализуют строки таблицы:
  - проверка начального состояния (`personAfterInit_hasNoLocation`);
  - посадка человека на разные локации из состояния `NO_LOCATION`;
  - запрет повторного `sitOn` на ту же самую локацию;
  - разрешённая смена локации на другую (`pavement → sand`, `sand → pavement`);
  - негативный сценарий `sitOn(null)`.

## 3. Конечный автомат местоположения `Child`

### 2.1. Состояния и события

- **Состояния:**

  - `NO_LOCATION` — ребёнок создан, но ещё не прыгнул.
  - `ON_SAND` — ребёнок находится на песке.

- **События/входы:**
  - `init(size)` — создание ребёнка с указанным размером.
  - `jumpOn(sand)` — прыжок на песок.

### 3.2. Таблица переходов для `Child`

| №   | Текущее состояние | Вход / условие                           | Следующее состояние | Выход / ожидаемый результат                                |
| --- | ----------------- | ---------------------------------------- | ------------------- | ---------------------------------------------------------- |
| 1   | (нет объекта)     | `init(size != null)`                     | `NO_LOCATION`       | `size` установлен, `location == null`                      |
| 2   | (нет объекта)     | `init(size == null)`                     | (исключение)        | `IllegalArgumentException("Size cannot be null")`          |
| 3   | `NO_LOCATION`     | `jumpOn(sand != null)`                   | `ON_SAND`           | `location == sand`                                         |
| 4   | `NO_LOCATION`     | `jumpOn(sand == null)`                   | (исключение)        | `IllegalArgumentException("Sand cannot be null")`          |
| 5   | `ON_SAND`         | `jumpOn(sand != null)` (повторный вызов) | `ON_SAND`           | `location` меняется/остаётся в последнем переданном `sand` |

### 3.3. Связь с тестами

- **Строка 1** — частично покрыта тестом `childHasNoLocationBeforeJumping` (проверка `location == null`).
- **Строка 3** — покрыта тестом `childrenJumpOnSand`.
- **Строки 2, 4, 5** — в текущем `AppTest` отсутствуют, их можно добавить как негативные/дополнительные проверки по таблице переходов.

## 4. Конечный автомат местоположения `Horse`

## 5. Конечный автомат (вычисление свежести) `Supply`

Фактически `Supply` больше похож на функцию от набора `ReinforcedFence`, но для методологии "таблиц переходов" можно трактовать это как простой автомат, где состояние определяется "общей свежестью".

### 5.1. Состояния и входы

- **Состояния:**

  - `FRESH` — все изгороди свежие.
  - `STALE` — хотя бы одна изгородь несвежая.

- **События:**
  - `init(fences)` — создание поставки с набором изгородей.
  - `getOverallFreshness()` — запрос состояния.

### 5.2. Таблица переходов для `Supply`

| №   | Текущее состояние | Вход / условие                  | Следующее состояние | Выход / ожидаемый результат                                          |
| --- | ----------------- | ------------------------------- | ------------------- | -------------------------------------------------------------------- |
| 1   | (нет объекта)     | `init(fences == null)`          | (исключение)        | `IllegalArgumentException("Supply must contain at least one fence")` |
| 2   | (нет объекта)     | `init(fences.isEmpty())`        | (исключение)        | то же, что строка 1                                                  |
| 3   | (нет объекта)     | `init(all fences FRESH)`        | `FRESH`             | последующий `getOverallFreshness() == Freshness.FRESH`               |
| 4   | (нет объекта)     | `init(есть хотя бы один STALE)` | `STALE`             | последующий `getOverallFreshness() == Freshness.STALE`               |

### 5.3. Связь с тестами

- **Строка 3** — покрыта тестом `suppliesAreFresh`.
- **Строка 4** — покрыта тестом `supplyWithOneStaleFenceIsNotFresh`.
- **Строки 1–2** — нет в текущем `AppTest`, стоит добавить негативные тесты.

## 6. Как использовать таблицы переходов для построения тестового покрытия

1. **Определить автоматы**: для вашей модели мы выделили минимум пять:

   - автомат эмоций `Person`,
   - автомат местоположения `Person`,
   - автомат местоположения `Child`,
   - автомат местоположения `Horse`,
   - автомат свежести `Supply`.

2. **Для каждого автомата выписать:**

   - набор состояний;
   - допустимые входы / события;
   - возможные комбинации "состояние + вход" и соответствующие им переходы.

3. **Сформировать таблицу переходов**: каждая строка таблицы — кандидат в отдельный тестовый случай.

4. **Построить тесты:**

   - для каждого уникального перехода (строки таблицы) написать тест, который:
     1. подготавливает объект в нужном текущем состоянии;
     2. прикладывает вход (вызов метода);
     3. проверяет следующее состояние и/или побочные эффекты (assert’ы).

5. **Покрыть как минимум:**

   - все "нормальные" переходы (например, прыжок ребёнка на песок, успешное перемещение лошади);
   - все "ошибочные" переходы (например, `null` в параметрах, пустой список изгородей).

6. **Сопоставить с существующими тестами**:
   - отметить, какие строки таблиц уже покрыты текущими тестами `AppTest`;
   - добавить недостающие тесты, чтобы добиться полного покрытия всех переходов, которые считаются допустимыми в вашей спецификации.

## 7. Что именно нужно переделать в вашем решении

1. **Оформить автоматы и таблицы переходов** — вы можете использовать данный файл `TRANSITION_TABLES.md` как основу, при необходимости дополнив или конкретизировав состояния/условия под ваши договорённости с преподавателем.
2. **Актуализировать тесты `AppTest`**:
   - явно связать каждый тест с одной или несколькими строками таблиц (хотя бы на уровне комментариев/описания теста);
   - добавить негативные тесты по строкам, где сейчас нет покрытий (см. секции 1.3, 2.3, 3.3, 4.3).
3. **На защите** пояснить:
   - какой автомат за что отвечает;
   - какие состояния и переходы вы выделили;
   - как из таблиц переходов были получены конкретные тест-кейсы.

Таким образом, ваша доменная модель из `task3` будет трактоваться как набор конечных автоматов, а тестовое покрытие — как реализация метода **"таблицы переходов"** для этих автоматов.
